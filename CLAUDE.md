## The idea
Run the **real dbt-core + dbt-duckdb, unmodified, entirely in the student's browser** via **Pyodide** (CPython compiled to WebAssembly), executing SQL against the **DuckDB Python client that Pyodide ships as a wasm wheel**.

- No backend. No login. **No install** - the student opens a URL and the genuine dbt engine runs client-side.
- This **replaces** transformation-lab's hand-written simulation engine (regex Jinja extraction, a deliberately reduced feature subset) with **the actual dbt**, so we get **near-total feature fidelity** instead of a maintained fake.
- Network at first load is fine (heavy download); **offline is explicitly NOT a requirement** (see D9).

### Why this is even possible (the unlock)
The DuckDB project compiles its **full Python client** to WebAssembly and the Pyodide team hosts it in their package repo (installable as plain `import duckdb`). That is *exactly* the interface `dbt-duckdb` expects (`import duckdb`, a `DuckDBPyConnection`). Because DuckDB is **embedded** (no socket, no network), it's the one warehouse that fits the browser sandbox. And because we run real CPython, we get **real Python Jinja2** - which erases the Jinja-fidelity long tail that would doom any reimplementation.

Source: https://duckdb.org/2024/10/02/pyodide

## Scope
**In scope:** dbt-duckdb **only**. Compilation + execution of the things that are plain SQL against an embedded DuckDB: `ref`/`source`, views/tables, **incremental**, **snapshots**, **seeds**, generic + singular **tests**, **macros**, real **Jinja**, contracts, docs, the semantic layer parsing, hooks, project/folder config. Python models too (they're just Python).

**Out of scope (by design):**
- Every other adapter - Snowflake, BigQuery, Databricks, Postgres, Redshift, etc. (they need live sockets; the whole point of choosing DuckDB is that it doesn't).
- Anything needing a **runtime-loaded DuckDB extension** (see Constraints).
- Offline operation.

## Architecture (target)
```
Monaco editor files ──► Pyodide in-memory FS (MEMFS): dbt_project.yml, profiles.yml, models/, seeds/, macros/
                            │
                            ▼
              dbt.cli.main.dbtRunner().invoke(["build", "--select", ...])   ← REAL dbt-core
                            │     (real Jinja2, real DAG resolution, real dbt-duckdb)
                            ▼
                DuckDB Python client (wasm, in-Pyodide) executes the SQL  - :memory:, threads:1
                            │
                            ▼
        Read back target/manifest.json + compiled/ + run results ──► DAG panel / terminal UI

Everything runs inside a Web Worker (Pyodide is single-threaded / GIL-bound); main thread only postMessages.
```

## Decisions log
| # | Decision | Rationale |
|---|---|---|
| **D1** | **Route B** - run real dbt-core in Pyodide. NOT a TypeScript rewrite of dbt (Route A, rejected). | Route A's Jinja2 parity is an endless long tail + you'd maintain a fake forever. Route B *is* dbt → fidelity for free. |
| **D2** | **dbt-duckdb only.** No other adapters. | DuckDB is embedded → no sockets → fits the browser. Other warehouses can't be HTML-native, period. |
| **D3** | **Build it single-agent / human-in-the-loop. Do NOT use a dynamic workflow to build it.** | This is a small, **sequential, critical-path integration** ("make it boot → iterate"), not a wide parallel translation. Workflows shine on breadth (cf. the Bun Zig→Rust port); they add only overhead here. Reserve **one** optional parallel pass for end-stage feature/lesson coverage (Step 3). |
| **D4** | Invoke dbt via the **programmatic `dbtRunner().invoke([...])`**, in-process. No CLI subprocess. | Subprocess/`os.exec` don't exist in Pyodide; `dbtRunner` returns result objects and writes `manifest.json`. |
| **D5** | **Stub `dbt-extractor`** (the Rust/tree-sitter static parser): ship a tiny Python module whose `py_extract_from_source` **always raises `ExtractionError`**, plus the `ExtractionError` class. | dbt is designed to **fall back to full Python-Jinja rendering** on any `ExtractionError`, and that fallback is the *high-fidelity* path. Avoids compiling a Rust wheel to wasm. Cost: slightly slower parse, zero feature loss. |
| **D6** | `threads: 1` in the duckdb profile. | Pyodide is single-threaded; avoids any path that would spawn real OS threads/processes. |
| **D7** | Pin to **one Pyodide release** and pick the **dbt-core version whose compiled deps all have matching Pyodide wheels** for that release. Expect a slightly-older dbt-core, not bleeding edge. | Compiled deps (duckdb, pydantic-core, msgpack, maybe pandas/pyarrow) must share one Python+emscripten ABI. dbt-core's pins must intersect what Pyodide ships. | **D8** | Disable telemetry: `send_anonymous_usage_stats: false`. | dbt's usage tracker uses `requests`; raw sockets don't work in Pyodide. It *should* fail gracefully, but disable it to be safe. |

| **D10** | Run Pyodide in a **Web Worker**. | Keeps the UI responsive; Pyodide is GIL-bound. |
| **D11** | **Stub `_multiprocessing`** (the missing C extension): a `threading`-backed `SemLock` (+ `sem_unlink`, `_posixshmem` stub). `SemLock.name=None` so `synchronize.py` skips resource-tracker registration (which would import `_posixshmem` and spawn a tracker subprocess). | This was the **#1 boot risk** and it bit exactly as predicted: `dbt-adapters/base/connections.py` does `from multiprocessing.synchronize import RLock`, which imports `_multiprocessing` unconditionally. | **D12** | **Make `concurrent.futures.ThreadPoolExecutor` synchronous** (inline `submit`, no worker threads). | Pyodide's default build cannot start OS threads at all. dbt-common's `MultiThreadedExecutor` (adapter ops: schemas, relation cache) inherits ThreadPoolExecutor → would die with "can't start new thread". | **D13** | **Patch dbt-duckdb's connection lifecycle**: `DuckDBConnectionWrapper.close()` does NOT close the cursor, and gains a working `rollback()`/`commit()` (issued as SQL on the cursor). | dbt connections are **thread-local**; a normal `--threads 1` run still has 2 threads (main `master` + 1 worker) so a task's `connection_named` release never disturbs master. ## Known constraints / non-goals
> **Companion docs:** [`ROADMAP.md`](./ROADMAP.md) = what's done / what's next. [`README.md`](./README.md) = how to run, **first-load footprint (~42 MB after the Step-2 networkx trim)**, and **how to bump dbt/Pyodide versions**.
- **No runtime-loaded DuckDB extensions.** The Pyodide duckdb build bundles only a few (parquet, json, icu, tpcds, tpch) and cannot load others at runtime. Don't author a lesson/model that needs `httpfs`, `spatial`, etc.
- **The duckdb `path:` must be a persistent file** (the default `/project/warehouse.duckdb` is). Since D30 lets a lesson/learner edit `profiles.yml`, note that **`:memory:` (or any non-file path) cannot work in this lab**: each `dbt` command is a *separate* `dbtRunner().invoke()`, so an in-memory db would be empty every command (models vanish between commands), and `query()`'s separate connection couldn't see dbt's memory anyway. This is inherent to the per-invoke model, not a bug - keep lesson paths file-backed. `query()`/`reset()` resolve the path verbatim from `profiles.yml` (relative paths resolve against the shared process cwd, so dbt and `query()` always agree).
- **Download bloat: ~~Pyodide's `networkx` build over-declares `matplotlib`+`numpy`~~ - FIXED (D21).** networkx now comes pure-python from PyPI, dropping the matplotlib→Pillow→contourpy→fonttools→kiwisolver→numpy stack (~11.6 MB). **Babel** (~10 MB CLDR locale data, via agate) is now the biggest single trim candidate left.
- **`dbt deps`** (fetching packages over git/HTTP) is the one mainstream feature that needs the network *and* a git client. Handle by **pre-bundling** packages into MEMFS, or proxying the fetch. Defer until needed. **Intercepted in the terminal (D29)** with a friendly "not available in the browser lab" message - alongside `docs serve` (listening socket) and `init` (interactive) - rather than letting dbt traceback/hang. `dbt docs generate` is NOT blocked (it just writes `target/`).
- **Heavy first load** - realistically ~30–60 MB (Pyodide + CPython + duckdb-wasm + pydantic-core + the wheel set) and a multi-second cold start. Acceptable per D9; mitigate with a loading state and lazy-loading behind the lesson.
- **pandas/pyarrow** may be pulled by dbt-duckdb's Python-model / external-parquet paths. Both exist in Pyodide but are heavyweight. Basic model runs shouldn't need them.
